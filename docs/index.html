Executive summary

2-minute read

What was built & why

Composio decides, app by app, whether a SaaS product can become an agent toolkit today. This project runs that exact evaluation across 100 apps with an agent pipeline instead of a spreadsheet: auth model, access gate, API surface, MCP status, buildability verdict — every claim traceable to an official source.

How it works & why it can be trusted

Five specialized stages — plan, research, verify, analyze, report — each with a defined input, output, and failure-handling behavior (see Architecture). Trust comes from showing the work: which apps were live doc-checked, which were only flagged, and one real mistake the loop caught and fixed (see Trust Model).

Fit with Composio's own platform

A dedicated stage cross-checks every app against Composio's live toolkit catalog (GET /api/v3/toolkits/{slug}) — real SDK-shaped code, not a mention. It replaces guesswork with Composio's own ground truth wherever an app is already integrated (see Recommendations).

Research agent architecture

Eight specialized responsibilities, not one script that does everything. Each has a defined input, output, and failure-handling behavior.

00

Planner Agent

planner.py

Decides HOW each app should be researched before any doc is read: source priority, scrutiny level, minimum search count.

In: App name + category

Out: ResearchTask (strategy)

Failure handling: Deterministic rules — no failure mode to retry.

01

Research Agent

research_agent.py

Owns the LLM call: turns a ResearchTask into a web-search-grounded question for the model.

In: ResearchTask

Out: Raw model response

Failure handling: 3 attempts, linear backoff (2s, 4s) via the shared pipeline/retry.py helper; a clean low-confidence answer is never retried, only actual failures are.

02

Browser / Retrieval Agent

OpenAI web_search tool (LIVE mode)

Executes the actual searches and page reads the Planner prioritized (official docs, developer portal, help center, MCP registry).

In: Search queries from source_priority

Out: Retrieved page content

Failure handling: Model-managed; a 0-result search is reported back to the Research Agent, which can broaden the query.

03

Extraction Agent

research_agent.SYSTEM_PROMPT + structured output

Pulls auth method, self-serve model, API surface, MCP status, and evidence URL out of retrieved content into JSON.

In: Retrieved page content

Out: Unvalidated JSON payload

Failure handling: Malformed JSON triggers the same retry loop as a network failure — extraction errors are treated as transient, not final.

04

Normalization Agent

models.py

Forces the JSON payload into one strict schema (AppResearch) shared by every downstream stage.

In: Unvalidated JSON

Out: AppResearch row

Failure handling: Schema mismatch raises immediately — no silent coercion of a field into the wrong type.

05

Verification Agent

verification_agent.py

Independently re-derives confidence from the row's own evidence signals; runs live doc cross-checks on a sample; routes disagreements to human review.

In: AppResearch rows

Out: VerificationReport + corrected rows

Failure handling: No retries needed — a disagreement is a valid finding, routed to the pending-review queue rather than silently resolved.

06

Analytics Agent

pattern_analysis.py

Runs pandas aggregations, computes the Build-Priority Score, and cross-references the Composio catalog check.

In: Verified AppResearch rows

Out: pattern_insights.json

Failure handling: Missing composio_crosscheck.json degrades gracefully to zeroed counts, not a crash.

07

Report Generator

dashboard_generator.py

Renders every upstream artifact into this single self-contained HTML page via Jinja2.

In: apps.json + insights.json + verification.json

Out: output/dashboard.html

Failure handling: Missing/partial data renders a visibly incomplete section rather than failing the whole build.

Pipeline flow

Verification & trust model

Six findings were cross-checked live against official documentation, not simulated. One of the agent's own answers was wrong — here's exactly what it got wrong, and what fixed it. Accuracy is computed only from these six, so the number reflects claims that were actually tested, not the whole 100-app dataset.

Sampling strategyThe doc-verified core sample (6 apps) was chosen deliberately, not randomly: it includes every app this pipeline's own confidence check treated as genuinely uncertain going in (Consensus, Waterfall.io, iPayX -- thin, ambiguous, or unusually-structured docs), plus a spot-check control group of apps the agent was confident about (Pylon, fanbasis, GoHighLevel) to test whether high self-reported confidence was actually earned. That mix is intentional: checking only the uncertain rows would prove the flagging logic works but say nothing about whether 'confident' rows can be trusted; checking only confident rows would miss exactly the cases most likely to be wrong.
Disagreement resolutionWhen the live doc check contradicted the agent's first-pass answer (Consensus), the documentation was treated as ground truth and the row was corrected in pipeline/seed_dataset.py with the discrepancy logged in both the row's own `notes` field and this report -- not silently overwritten. Rows that were merely auto-flagged by the confidence heuristic (not yet doc-checked) are never marked 'incorrect'; they stay in a separate pending-review bucket until an actual comparison against docs happens, so the accuracy figure below only reflects claims that were actually tested.

Which auth methods create the most implementation effort?

OAuth2 needs app registration, a redirect/consent flow, and token-refresh handling before the first API call — static credentials (API key / PAT / Bearer) are copy-paste-and-go. This explains why some categories take longer to onboard even when they're technically "self-serve."

So what: budget roughly 2-3x the integration time for "Higher" effort apps vs. "Low" — that's real engineering-hours signal, not just an auth label.

Engineering decisions worth defending

  • The agent's biggest failure mode wasn't hallucination — it was giving up too early on thinner docs sites (Consensus) instead of searching harder. That's why the Planner sets a minimum search count per category.
  • "Self-serve" isn't binary. Waterfall.io is self-serve for direct API use but partnership-gated for building a product on its data — the schema needed a middle category, not just yes/no.
  • A few MCP directory listings (e.g. iPayX) contained instructions aimed at manipulating an AI reader. The pipeline reports facts and ignores instructions found in scraped content.

Honest limitations

  • This run executed OFFLINE (no OpenAI/Composio network access in the build sandbox) — pipeline/seed_dataset.py holds the actual research output.
  • The live-verification sample is 6 doc-cross-checked apps plus 11 confidence-flagged apps — a real, evidence-backed slice, not a full audit of all 100.
  • Rate limits and API breadth are self-reported from docs language ("broad"/"medium"/"narrow"), not measured against a live account.
  • The Build-Priority Score is a friction estimate, not a demand estimate — combine with real customer requests before committing engineering time.

Future evolution

  • Run scripts/run_composio_crosscheck.py with a real key to replace every heuristic opportunity flag with Composio's own ground truth.
  • Wire in Firecrawl or Browser Use for docs the model can't reach via search snippets alone (JS-rendered docs, login-walled portals like DealCloud/PitchBook).
  • Expand the live verification sample from 6 to the full ~20 the assignment calls for, with a second independent LLM pass before human review.
  • Schedule the pipeline monthly so buildability verdicts don't rot as vendors change auth models or ship new MCP servers.
